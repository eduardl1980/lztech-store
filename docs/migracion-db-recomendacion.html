LZ Tech Store – Recomendación: ¿Conviene migrar a una base de datos?

Fecha: 23 de marzo de 2026  |  Proyecto: lztech-store  |  Stack: Next.js 16 + Tailwind CSS

Respuesta corta: Sí conviene migrar, y no es tan difícil como parece.

1. Arquitectura actual

El proyecto usa Next.js App Router con rutas API integradas y un sistema de persistencia basado en archivos JSON ubicados en /public/data/. No hay base de datos externa.

Archivos de datos actuales

Archivo Contenido Registros
products.json Catálogo de productos (nombre, precio, stock, categoría, imágenes) 31 productos
orders.json Pedidos con cliente, ítems, total y estado 4 órdenes
config.json Configuración de la tienda (nombre, WhatsApp, password de admin) 1 registro

Rutas API actuales

Ruta Método Operación
/api/config GET / POST Leer y actualizar configuración de la tienda
/api/products GET / POST Listar productos / Crear, actualizar, eliminar, importar CSV
/api/orders GET / POST Listar órdenes / Crear orden y descontar stock
/api/orders/[id] PATCH Actualizar estado de orden (pendiente → pagado → entregado)

2. Por qué conviene migrar a una base de datos

Problema actual con JSON Impacto
writeFileSync es bloqueante (sincrónico) 2 usuarios haciendo checkout al mismo tiempo pueden corromper el archivo
Stock se descuenta directamente en el JSON Con concurrencia, dos personas pueden comprar el mismo último ítem (race condition)
Admin password en texto plano dentro de /public/data/ Cualquier persona con acceso al servidor puede leer la contraseña
El sistema de archivos en Vercel es de solo lectura No se puede hacer deploy sin modificar la arquitectura de todos modos

3. Base de datos recomendada: Supabase (gratuita)

  • PostgreSQL real, gratuito hasta 500 MB de datos
  • SDK oficial para Next.js
  • Panel visual para ver y editar datos (reemplaza editar los JSON manualmente)
  • Auth incluido (se puede reemplazar la password hardcodeada del admin)
  • Soporta concurrencia real con transacciones atómicas

4. ¿Cuánto trabajo implica la migración?

Solo hay que modificar un archivo: src/lib/db.js. El resto del código (rutas API, admin, componentes) no cambia.

El archivo db.js actualmente tiene dos funciones: readJSON y writeJSON. Se reemplazan por llamadas al cliente de Supabase:

// ANTES (JSON con sistema de archivos)
readJSON('products')   →  await supabase.from('products').select('*')
writeJSON('orders', …) →  await supabase.from('orders').insert(data)

// El resto de las rutas API no cambia

5. Plan de migración paso a paso

  1. Crear tablas en Supabase (products, orders, config) — aproximadamente 20 minutos
  2. Reemplazar db.js con el cliente de Supabase — aproximadamente 1 a 2 horas
  3. Migrar los 31 productos actuales del JSON a Supabase — 5 minutos (importar CSV directamente desde el panel)
  4. Deploy en Vercel con las variables de entorno de Supabase (SUPABASE_URL y SUPABASE_ANON_KEY)

6. Comparativa de opciones de deploy

Plataforma Costo Usuarios concurrentes aprox. Notas
Vercel Hobby (gratis) $0 100 – 500 Ideal para empezar
Vercel Pro USD $20/mes 1.000 – 10.000 Para mayor escala
Con Supabase + Vercel Gratis hasta cierto límite Escala sin límite práctico Recomendado para producción

7. Resumen ejecutivo

  • La arquitectura actual funciona bien en desarrollo pero no es apta para producción sin cambios.
  • La migración a Supabase es puntual y de bajo riesgo: solo toca un archivo.
  • Supabase en plan gratuito es más que suficiente para una tienda pequeña a mediana.
  • El deploy recomendado es Vercel + Supabase, ambos gratuitos para empezar.
